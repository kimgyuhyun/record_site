# 전적 갱신 작업 큐 설계 & 기술 선택 이유

> 메인 README 에 붙여넣을 수 있는 초안. "전적 갱신"을 동기 처리에서 **작업 큐 기반 비동기 + 진행률 추적**으로 개선한 배경과 트레이드오프를 정리한다.

## 문제

"전적 갱신"은 최근 매치 최대 20개 × (매치 상세 + 타임라인) 호출이라, 신규 매치가 많으면 한 요청에 **수십 번의 Riot API 호출이 순차로** 일어난다. Riot 개발 키 한도(20 req/s, 100 req/2분) 때문에 throttle 도 필요해 **10~30초**까지 걸린다.

이를 동기로 처리하면:

- 사용자가 응답을 받을 때까지 **HTTP 요청이 그만큼 블로킹**된다.
- 갱신 트래픽과 백그라운드 매치망 크롤러가 **같은 레이트리밋 예산을 공유**해, 동시 요청이 몰리면 서로의 대기를 키운다.

→ 요청을 **작업 큐에 넣고 즉시 응답**하고, **백그라운드 워커가 수집**하며, 프론트는 **진행률을 폴링**하는 구조로 바꿨다.

## 아키텍처

```
[1] POST /api/matches/refresh?puuid=X
      → SET refresh:lock:{puuid} {jobId} NX EX 120   (중복 갱신 락)
          성공 → job:{jobId} 해시(PENDING) 생성 → LPUSH refresh:queue {jobId} → 즉시 응답
          실패 → 락이 가리키는 기존 jobId 의 상태를 그대로 응답
      ← { jobId, status: "PENDING", total, done }

[2] 워커(전용 스레드 1개): BRPOP refresh:queue
      - status = PROCESSING
      - 최근 매치 ID 20개 조회(호출 1회)
      - DB에 이미 있는 매치는 스킵(증분 수집) → 신규 수 = total
      - 신규 매치만 하나씩 상세 수집(공유 레이트리밋 통과) → DB 저장 → done++
      - status = DONE  (실패 시 FAILED + 락 해제)

[3] GET /api/matches/refresh-jobs/{jobId} → { status, total, done }
      프론트가 2.5초 간격 폴링 → DONE 이면 소환사/매치 재조회
```

### Redis 키

| 키 | 타입 | 용도 | TTL |
|---|---|---|---|
| `refresh:lock:{puuid}` | String(jobId) | 중복 갱신 차단 + 직후 쿨다운 | 120s |
| `refresh:queue` | List | 대기 jobId (LPUSH/BRPOP) | — |
| `job:{jobId}` | Hash | `status/total/done/puuid` | 600s |

## 기술 선택 & 트레이드오프

### 1. 왜 작업 큐(비동기)인가
동기 응답은 구현이 단순하지만 사용자를 수십 초 잡아둔다. 큐로 분리하면 요청은 즉시 끝나고(서버 스레드 점유 X), 무거운 수집은 워커가 자기 페이스로 처리한다. 진행률(`total/done`)을 노출해 "얼마나 남았는지" UX 도 개선된다.

### 2. 왜 Redis List + BRPOP / 전용 워커 스레드인가
- 이미 **Redis 를 캐시로 쓰고 있어** 별도 인프라(Kafka/RabbitMQ) 추가 없이 `List` 하나로 큐를 만들 수 있다.
- 소비는 `@Scheduled` **폴링** 대신 **BRPOP(블로킹)** 을 택했다. 큐가 비면 워커가 잠들어 있다가 작업이 들어오는 즉시 깨어나므로, **idle CPU 낭비가 없고 처리 지연이 거의 0**이다. 대신 블로킹 호출이라 `SmartLifecycle` 로 **전용 스레드 1개를 직접 기동/종료**한다.
- 워커가 1개라 동시성 제어(작업 클레임 락)가 필요 없다. 소수 사용자 규모에 맞춘 의도된 단순화다.

### 3. 레이트리밋터는 새로 만들지 않고 재사용
요구안에는 "Redis 토큰버킷 레이트리밋터"가 있었지만, 이 프로젝트엔 이미 **공유 슬라이딩 윈도우 리미터**(`RiotApiRateLimiter`)와 **429 Retry-After 백오프**(`RiotApiClientInterceptor`)가 있다. 모든 Riot 호출이 이 인터셉터를 통과하므로 워커도 자동으로 throttle 된다.
- **단일 백엔드 프로세스에서는 in-memory 리미터가 더 정확하고 단순**하다. Redis 토큰버킷은 **여러 워커 프로세스가 예산을 공유해야 할 때** 의미가 있다 — 지금 규모엔 과한 추상화라 도입하지 않았다.
- (확장) 워커를 여러 인스턴스로 수평 확장한다면 그때 Redis 기반 분산 리미터로 교체하면 된다.

### 4. 중복 방지 락이 쿨다운을 겸한다
`SET NX EX 120` 으로 같은 puuid 의 동시/연타 갱신을 막는다. **성공해도 락을 즉시 지우지 않고 TTL(120s)로 자연 만료**시켜, 그 사이 재요청은 기존 작업 상태를 돌려받는다(= 쿨다운). **실패 시에는 즉시 락을 해제**해 바로 재시도할 수 있게 했고, 워커가 죽어도 TTL 로 자동 해제된다. 락 해제는 `GET == 내 jobId` 일 때만 지우는 **compare-and-delete(Lua)** 로 남의 락을 건드리지 않는다.

### 5. 멱등성 · 증분 · 부분 실패
- **증분 수집**: 매치 ID 목록을 받아 `findExistingMatchIds` 로 DB에 없는 것만 상세 조회한다(첫 조회만 풀 수집).
- **멱등성**: 매치는 matchId 로 식별·중복 저장 방지된다. 같은 작업을 다시 돌려도 신규만 추가된다.
- **부분 실패 허용**: 특정 매치 수집이 실패하면 그 매치만 스킵하고 진행률은 전진시킨다. 한 건의 실패가 전체 갱신을 막지 않는다.

### 6. 기존 백그라운드 크롤러 큐와는 분리 유지
매치망 점진 수집 크롤러는 별도의 **DB 기반 큐**(`pending_summoner_refresh`)를 단일 `@Scheduled` 워커로 처리한다. 사용자 갱신(우선·즉시 반응)과 크롤러(백그라운드 표본 수집)는 성격이 달라 **큐를 합치지 않고 분리**했다.

## 한계 & 향후 개선

- **락 TTL < 수집 시간**: 신규 매치가 비정상적으로 많아 수집이 120초를 넘기면 락이 먼저 만료돼 중복 작업이 생길 수 있다(매치 20개 한도에선 사실상 발생하지 않음). 필요 시 처리 중 락을 연장하면 된다.
- **진행률 vs 트랜잭션**: 수집은 한 트랜잭션에서 커밋되므로, 진행률(`done`)이 올라가도 매치는 작업 완료 시점에 한꺼번에 조회 가능해진다. 프론트는 DONE 후 재조회한다.
- **수평 확장**: 워커를 다중화하면 (1) 분산 레이트리밋(Redis 토큰버킷), (2) 작업 클레임 락이 필요하다. 현재는 단일 워커 전제다.
- **(선택)** 진행률을 폴링 대신 SSE 로 push, 큐를 RabbitMQ 로 교체하는 분기도 가능하다 — 포트폴리오 확장 포인트.
